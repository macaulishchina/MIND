## 1. 目标与动机

### 要解决的问题

当前 MIND 的提取流程是：

```
自然语言对话
  → LLM 提取自然语言事实 [{text, confidence}]
  → LLM 归一化为 FactEnvelope (JSON)
  → 写入向量库 + 关系库
```

这有三个结构性缺陷：

1. **两步 LLM 调用**（提取 + 归一化），成本高、信息在中间步骤丢失
2. **JSON 输出脆弱**，格式错误导致整条丢失
3. **只能表达事实**，希望、条件、转述、谎言、不确定性等信息被丢弃

### 语义翻译层的三个目标

| # | 目标 | 策略 |
|---|------|------|
| 1 | **不丢信息** | 引入 Frame 对象，覆盖事实/希望/条件/谎言/转述/否定/不确定等 |
| 2 | **压缩 token** | 紧凑的函数式语法，比 JSON 输出减少 ~30-45% token（简单场景接近 45%，复杂场景约 30%） |
| 3 | **直接映射到结构化存储** | 每条语句用正则+括号匹配栈解析，四级容错保证不丢信息 |

### 一句话定义

> 不是直接把自然语言变成"记忆文本"，而是先变成一种**紧凑的语义符号层**。LLM 一步完成翻译，下游用正则+括号匹配栈完成解析入库。

## 2. 设计原则：语法封闭 × 词汇开放

这套系统的核心设计哲学：

| 维度 | 策略 |
|------|------|
| **语法结构** | 只有 **5 种句式**，LLM 必须从中选择，程序用正则+括号匹配栈解析 |
| **谓词/关系词** | 提供一个**种子词表**（~60 词），但 LLM 可以**自创新词**，新词入库后可复用 |

为什么这样设计：

- **语法封闭** → 解析器永远稳定，不需要随词汇更新而更新
- **词汇开放** → 表达力不受限，LLM 可以用最精确的词描述语义
- **新词有注册机制** → 不是无序扩展，而是有声明、有定义、可复用的受控扩展

---

## 3. 四类一等公民

不把"事实文本"当作最终单位，而把四类对象当作一等公民：

| 对象 | 符号 | 含义 | 类比 |
|------|------|------|------|
| **Ref** | `@id` | 实体引用——"谁/什么" | KG 中的 node |
| **Prop** | `$id` | 命题——"什么事/什么属性" | KG 中的 edge |
| **Frame** | `$id` | 对命题的态度/包装——"希望/条件/转述/撒谎" | Reified meta-edge |
| **Evidence** | `ev()` | 来源、置信度、原文——"凭什么这么说" | Provenance |

关键设计点：

- **实体和关系分开**：`Tom` 是 Ref，`friend(@self, Tom)` 是 Prop
- **关系和属性统一为命题**：`friend(a, b)` 和 `occupation(a, x)` 结构相同
- **Frame 和 Prop 语法同构**：都是 `$id = pred(args...)`，靠词表分类区分
- **希望、条件、撒谎不是"附注"**，而是对命题的进一步建模

---

## 4. 5 种固定句式

每条输出**必须且只能**是以下 5 种之一：

```
① REF    @id = ref_expression                        # 声明实体
② PROP   $id = predicate(arg, arg, ...)              # 断言命题
③ FRAME  $id = frame_pred(arg, arg, ...)             # 框架（与②语法同构）
④ EV     ev($id, conf=N, src="turn_X", span="...")   # 证据
⑤ NOTE   note($id, "free text")                      # 注释/兜底
```

加上注释行：

```
⑥ COMMENT  # 任意注释文字                             # 被解析器忽略
```

**PROP 和 FRAME 的语法完全相同**（`$id = word(args...)`）。区分它们不靠语法，靠谓词在词表中的分类。这意味着**解析器只需要一条规则处理两者**。

**NOTE 是安全阀**——LLM 遇到无法用前 4 种表达的信息时，不丢弃，写进 NOTE。

## 5. 实体引用（Ref）详解

### 语法

```
@id = ref_expression
```

### ref_expression 的 4 种形式

| 形式 | 含义 | 示例 |
|------|------|------|
| `@self` | 当前 owner（用户本人） | `@s = @self` |
| `@local/TYPE("key")` | owner 语境下的本地实体 | `@t = @local/person("tom")` |
| `@world/TYPE("key")` | 外部世界实体 | `@k = @world/city("tokyo")` |
| `_:id` | 未命名实体（blank node） | `@u = _:p1` |

### alias 支持

同一个实体可能有多个称呼（"Tom" / "小汤" / "Tommy"）。REF 支持可选的 alias 声明：

```
@t = @local/person("tom", alias=["小汤", "Tommy"])
```

alias 在解析时写入 refs 表，用于跨对话的实体合并。

### TYPE 的种子值

```
person, place, org, brand, event, object, animal, food, concept, time
```

LLM 可以自创新 type（如 `software`, `disease`），遵循与谓词相同的自创规则。

### 设计要点

- **先声明再使用**：所有 `@id` 必须在使用前声明
- **`@self` 不需要声明**：它是隐式存在的
- **key 保留原文**：中文名用中文，英文名用英文，不做翻译
- **TYPE 归一化为英文**：`person` 不写 `人物`

## 6. 命题与框架（Prop / Frame）详解

### 语法（两者同构）

```
$id = predicate(arg, arg, ...)
```

### arg 的 6 种类型

| 类型 | 语法 | 示例 |
|------|------|------|
| 实体引用 | `@id` | `@t` |
| 命题/框架引用 | `$id` | `$p1` |
| 字面字符串 | `"text"` | `"football player"` |
| 数值 | `number` | `42`, `0.8` |
| 列表 | `[a, b, c]` | `["中文", "英语", "日语"]` |
| 内联嵌套 | `pred(args)` | `neg(eat(@s, "pork"))` |

### 硬约束

- **内联嵌套最多 2 层**：`say(@m, must(@s, visit(@s, @m)))` 可以，但不允许再嵌套第 4 层
- **如需更深嵌套**，必须先声明中间 `$id`
- **列表内不允许嵌套列表**

这确保解析器永远不需要处理递归嵌套。

### Prop vs Frame 的区分

两者语法相同，靠谓词在 `vocab_registry` 中的 category 区分：

| category | 语义 | 示例谓词 |
|----------|------|---------|
| `prop` | 关于世界的断言 | `friend`, `occupation`, `live_in`, `like` |
| `frame` | 对命题的态度/逻辑包装 | `believe`, `hope`, `say`, `if`, `cause`, `neg` |
| `qualifier` | 修饰其它命题的维度 | `time`, `degree`, `quantity`, `frequency` |

这个区分**不影响解析**，只影响下游查询和推理。

### 参数位置语义

参数的语义靠**位置 + 谓词的 arg_schema** 确定：

```
friend(person_a, person_b)      # arg_schema = "person_a,person_b"
say(speaker, content)            # arg_schema = "speaker,content"
give(giver, receiver, object)    # arg_schema = "giver,receiver,object"
time(target_prop, time_value)    # arg_schema = "target,value"
```

arg_schema 记录在 `vocab_registry` 表中。解析器按位置存储，查询时查 schema 获取语义角色。

## 7. 证据（Evidence）详解

### 语法

```
ev($target_id, conf=N, src="turn_X", span="原文片段")
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `$target_id` | 是 | 指向哪条命题/框架 |
| `conf` | 是 | 置信度，0.0~1.0 |
| `src` | 否 | 来源标识，如 `"turn_7"` |
| `span` | 否 | 原文中支持这条命题的片段 |
| `residual` | 否 | 无法形式化但不想丢的残余语义 |

### 置信度语义

| 值 | 含义 |
|----|------|
| `1.0` | 用户直接明确陈述 |
| `0.8` | 明确但有轻微压缩 |
| `0.6` | 可信转述/轻微模糊 |
| `0.4` | 间接推断 |
| `0.2` | 高度不确定 |

### 设计要点

- **每条 $id 都应有至少一条 ev()**——没有来源的命题是悬空的
- **一条 $id 可以有多条 ev()**——多段对话都提到同一事实
- **residual 是最后的信息保全手段**——语气、讽刺等无法形式化的语义放这里

## 8. 注释（Note）详解

### 语法

```
note($target_id, "free text content")
```

### 两种用途

#### 用途 1：新谓词声明

LLM 自创新谓词时，必须附带一条半结构化 note：

```
note($p1, "NEW_PRED obsessed_with | frame | experiencer,target | extreme liking beyond habit")
```

格式：`NEW_PRED word | category | arg_schema | definition`

解析器检测 `NEW_PRED` 前缀，自动提取谓词元数据写入 `vocab_registry`。

#### 用途 2：兜底自由文本

无法用 Ref/Prop/Frame/Ev 表达的信息：

```
note($p1, "用户说这句话时笑了")
note($f1, "语气是半开玩笑的，不确定是否认真")
```

### 设计要点

- NOTE 不是垃圾桶——它是**信息保全的最后一道防线**
- 如果一段对话中大量信息只能靠 NOTE 表达，说明词表或 Frame 模型需要扩展

## 9. 词汇系统：种子词表 + LLM 自创

### 种子词表

#### PROP 谓词（~40 个）

| 类别 | 种子词 |
|------|--------|
| **关系** | `friend`, `mother`, `father`, `brother`, `sister`, `spouse`, `partner`, `child`, `cousin`, `coworker`, `boss`, `mentor`, `student`, `roommate`, `neighbor`, `classmate`, `teammate`, `client`, `landlord`, `doctor`, `pet` |
| **属性** | `name`, `age`, `occupation`, `location`, `workplace`, `language`, `education`, `nationality` |
| **行为/状态** | `like`, `dislike`, `habit`, `hobby`, `skill`, `own`, `use`, `eat`, `drink`, `speak`, `live_in`, `work_at`, `study_at` |
| **事件** | `plan`, `event`, `buy`, `visit`, `meet`, `resign`, `marry`, `move`, `start`, `stop`, `birthday`, `gift` |

#### FRAME 谓词（~20 个）

| 类别 | 种子词 | 含义 |
|------|--------|------|
| **认知** | `believe`, `doubt`, `know`, `uncertain` | 信念态度 |
| **意愿** | `hope`, `want`, `intend` | 意图愿望 |
| **言语** | `say`, `recommend`, `ask`, `promise` | 转述类 |
| **逻辑** | `if`, `cause`, `because` | 条件因果 |
| **规范** | `must`, `permit`, `should` | 义务许可 |
| **真值** | `neg`, `lie`, `joke`, `retract_intent`, `correct_intent` | 否定/修正意图 |
| **情感** | `emotion` | 情感状态 |
| **决策** | `decide`, `defer`, `undecided` | 决策状态 |

#### QUALIFIER 谓词（~6 个）

| 种子词 | 含义 | 示例 |
|--------|------|------|
| `time` | 时间锚点 | `time($p1, "2025")`, `time($p1, "recent")` |
| `degree` | 程度 | `degree($p1, "slight")` |
| `quantity` | 数量 | `quantity($p1, "5km/day")` |
| `frequency` | 频率 | `frequency($p1, "twice_a_week")` |
| `duration` | 持续时长 | `duration($p1, "3_years")` |
| `location` | 地点修饰命题 | `location($p1, @place)` |

#### REF TYPE（~10 个）

```
person, place, org, brand, event, object, animal, food, concept, time
```

### LLM 自创规则

当种子词表不够精确时，LLM **应该**自创新词。规则：

| 规则 | 说明 |
|------|------|
| **格式** | 小写英文，下划线分隔：`infatuated`, `passive_aggressive`, `binge_watch` |
| **声明** | 第一次使用新词时，必须附带 `note($id, "NEW_PRED ...")` |
| **精确优先** | 鼓励精确描述——`obsessed_with` 比 `like` 更准确时优先用前者 |
| **定义里给 arg_schema** | 方便解析器自动注册参数签名 |

#### 自创示例

```text
# 种子词 "like" 不够精确
$p1 = obsessed_with(@s, "咖啡")
note($p1, "NEW_PRED obsessed_with | frame | experiencer,target | extreme degree of liking")

# 需要表达 "默许"
$f1 = acquiesce(@s, $p2)
note($f1, "NEW_PRED acquiesce | frame | agent,content | passive acceptance without explicit agreement")

# "friend" 不够精确
$p1 = childhood_friend(@s, @t)
note($p1, "NEW_PRED childhood_friend | prop | person_a,person_b | friends since childhood")
```

### 词汇生命周期

```
LLM 自创新词 + note 声明
    ↓
解析器检测 NEW_PRED → embedding 碰撞检测
    ↓ sim > 0.85 → 提示 LLM 考虑复用已有近义词
    ↓ sim ≤ 0.85 → 写入 vocab_registry (status=candidate)
    ↓
使用频次 ≥ 3 且跨 ≥ 3 个 turn → 晋升 established
    ↓
使用频次 ≥ 10 + 手动确认 → 晋升 seed
    ↓
90 天未使用 → dormant
    ↓
180 天未使用 → archived
```

**生命周期状态**：

| 状态 | 条件 |
|------|------|
| `candidate` | 刚创建，< 3 次独立使用 |
| `established` | ≥ 3 次使用，跨 ≥ 3 个 turn |
| `seed` | ≥ 10 次，手动确认后晋升 |
| `dormant` | 90 天未使用 |
| `archived` | 180 天未使用 |

**碰撞检测**：新词入库前，embedding 查询 `vocab_registry` 中已有词。如果最近义词 sim > 0.85，解析器记录告警，提示 LLM 在后续提取中考虑复用。

## 10. 形式文法（EBNF）

```ebnf
program     = { line "\n" } ;
line        = ref_stmt | prop_stmt | ev_stmt | note_stmt | comment ;

(* ① REF *)
ref_stmt    = "@" id "=" ref_expr ;
ref_expr    = "@self"
            | "@local/" type "(" quoted [ "," "alias=" list ] ")"
            | "@world/" type "(" quoted [ "," "alias=" list ] ")"
            | "_:" id ;

(* ②③ PROP & FRAME — 语法同构 *)
prop_stmt   = "$" id "=" predicate "(" arg_list ")" ;
arg_list    = arg { "," arg } ;
arg         = ref_arg                           (* @id          *)
            | prop_arg                          (* $id          *)
            | quoted                            (* "literal"    *)
            | number                            (* 42, 0.8      *)
            | list                              (* [a, b, c]    *)
            | predicate "(" arg_list ")" ;      (* inline, 最多2层 *)
ref_arg     = "@" id ;
prop_arg    = "$" id ;
list        = "[" arg { "," arg } "]" ;

(* ④ EVIDENCE *)
ev_stmt     = "ev(" "$" id { "," kv_pair } ")" ;
kv_pair     = key "=" ( quoted | number ) ;

(* ⑤ NOTE *)
note_stmt   = "note(" "$" id "," quoted ")" ;

(* COMMENT *)
comment     = "#" { any_char } ;

(* Atoms *)
predicate   = letter { letter | "_" | digit } ;
type        = letter { letter | "_" } ;
id          = letter { letter | digit | "_" } ;
key         = letter { letter | "_" } ;
quoted      = '"' { any_char_except_dquote } '"' ;
number      = [ "-" ] digit { digit } [ "." digit { digit } ] ;
```

### 硬约束清单

| 约束 | 规则 |
|------|------|
| 每行恰好一条语句 | 不允许多条语句在同一行 |
| 内联嵌套最多 2 层 | `pred(pred(pred(...)))` 可以，4 层起必须用 `$id` 拆解 |
| 列表内不嵌套列表 | `["a", "b"]` 可以，`[["a"], ["b"]]` 不行 |
| 先声明后使用 | `@id` 必须在 REF 行中声明后才能在 PROP/FRAME 中引用 |
| `@self` 免声明 | 它是隐式存在的 |

## 11. 完整示例集

### 11.1 简单关系+属性

**自然语言**：`我朋友 Tom 是足球运动员`

```text
@s = @self
@t = @local/person("tom")
$p1 = friend(@s, @t)
$p2 = occupation(@t, "football_player")
ev($p1, conf=1.0, src="turn_1")
ev($p2, conf=1.0, src="turn_1")
```

### 11.2 未命名第三方

**自然语言**：`我有个朋友是足球运动员`

```text
@s = @self
@u = _:p1
$p1 = friend(@s, @u)
$p2 = occupation(@u, "football_player")
ev($p1, conf=1.0, src="turn_1")
ev($p2, conf=1.0, src="turn_1")
```

### 11.3 多值属性

**自然语言**：`我会说中文、英语和一点点日语`

**推荐写法（展开为多条语句）**：

```text
@s = @self
$p1 = speak(@s, "中文")
$p2 = speak(@s, "英语")
$p3 = speak(@s, "日语")
$p4 = degree($p3, "slight")
ev($p1, conf=1.0, src="turn_1")
ev($p2, conf=1.0, src="turn_1")
ev($p3, conf=1.0, src="turn_1")
ev($p4, conf=0.8, src="turn_1", span="一点点")
```

> **设计决议**：列表 `[a, b, c]` 语法在 EBNF 中保留兼容，但 Prompt 推荐展开为多条语句。展开后每条有独立 `ev()`、独立 qualifier，行级独立解析。

### 11.4 希望

**自然语言**：`我希望 Tom 来东京`

```text
@s = @self
@t = @local/person("tom")
@k = @world/city("tokyo")
$p1 = come(@t, @k)
$f1 = hope(@s, $p1)
ev($f1, conf=0.9, src="turn_1")
```

### 11.5 条件

**自然语言**：`如果明天不下雨，我打算去跑步`

```text
@s = @self
$p1 = plan(@s, "running")
$f1 = if(neg(rain("tomorrow")), $p1)
ev($f1, conf=0.9, src="turn_1")
```

### 11.6 转述

**自然语言**：`Mike 说 Tom 住在东京`

```text
@m = @local/person("mike")
@t = @local/person("tom")
@k = @world/city("tokyo")
$p1 = live_in(@t, @k)
```

## 12. 数据库映射

### 关系型数据库 Schema

```sql
-- ========== 会话与轮次 ==========

-- 对话会话
CREATE TABLE conversations (
    id          TEXT PRIMARY KEY,         -- UUID
    started_at  TIMESTAMPTZ DEFAULT NOW(),
    metadata    JSONB DEFAULT '{}'        -- 可选：来源、摘要等
);

-- 对话轮次
CREATE TABLE turns (
    id          SERIAL PRIMARY KEY,
    conv_id     TEXT NOT NULL REFERENCES conversations(id),
    turn_index  INT NOT NULL,             -- 该会话内的第几轮
    role        TEXT NOT NULL,            -- "user" | "assistant"
    content     TEXT NOT NULL,            -- 原始对话文本
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (conv_id, turn_index)
);

-- 提取批次（每次独立提取的元数据）
CREATE TABLE extraction_batches (
    id          TEXT PRIMARY KEY,         -- UUID
    conv_id     TEXT NOT NULL REFERENCES conversations(id),
    turn_start  INT NOT NULL,             -- 本批次覆盖的起止 turn_index
    turn_end    INT NOT NULL,
    model       TEXT,                     -- 使用的 LLM 模型
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- ========== 核心语义表 ==========

-- 实体引用
CREATE TABLE refs (
    id          TEXT PRIMARY KEY,         -- 全局唯一：按 (scope, ref_type, key) upsert
    scope       TEXT NOT NULL,            -- "self" | "local" | "world" | "blank"
    ref_type    TEXT,                     -- "person" | "place" | ...
    key         TEXT,                     -- "tom"
    aliases     JSONB DEFAULT '[]',       -- ["小汤", "Tommy"]
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 命题 + 框架（同表）
CREATE TABLE statements (
    id             TEXT PRIMARY KEY,         -- 全局唯一：batch_id + "_" + LLM输出的本地id
    batch_id       TEXT REFERENCES extraction_batches(id),
    predicate      TEXT NOT NULL,            -- "friend"
    args           JSONB NOT NULL,           -- ["@s", "@t"]（全部扁平化，无内联谓词）
    category       TEXT,                     -- "prop" | "frame" | "qualifier"
    superseded_by  TEXT REFERENCES statements(id),
    is_current     BOOLEAN DEFAULT TRUE,
    created_at     TIMESTAMPTZ DEFAULT NOW()
);

-- 反向索引：statement 引用了哪些 ref/prop
CREATE TABLE stmt_refs (
    stmt_id     TEXT NOT NULL,
    position    INT NOT NULL,
    ref_id      TEXT NOT NULL,            -- "@t" 或 "$p1"
    PRIMARY KEY (stmt_id, position)
);

-- 证据
CREATE TABLE evidence (
    id          SERIAL PRIMARY KEY,
    target_id   TEXT NOT NULL,            -- statements.id
    conf        REAL NOT NULL,
    src         TEXT,                     -- "turn_7" → 可关联 turns 表
    span        TEXT,
    residual    TEXT,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 注释
CREATE TABLE notes (
    id          SERIAL PRIMARY KEY,
    target_id   TEXT NOT NULL,
    content     TEXT NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 词汇注册表
CREATE TABLE vocab_registry (
    word         TEXT PRIMARY KEY,
    category     TEXT NOT NULL CHECK (category IN ('prop','frame','qualifier','ref_type')),
    arg_schema   TEXT,                    -- "agent,location"
    definition   TEXT NOT NULL,
    source       TEXT DEFAULT 'seed',     -- "seed" | "llm_created"
    usage_count  INT DEFAULT 0,
    status       TEXT DEFAULT 'candidate'
                 CHECK (status IN ('candidate','established','seed','dormant','archived')),
    first_seen_turn INT,
    last_used_turn  INT,
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    last_used    TIMESTAMPTZ
);

-- 共指消解记录
CREATE TABLE coreference (
    id          SERIAL PRIMARY KEY,
    source_expr TEXT,                     -- "他", NULL=零指代
    resolved_to TEXT NOT NULL,            -- refs.id
    turn_id     INT REFERENCES turns(id),
    confidence  REAL NOT NULL,
    method      TEXT                      -- "focus_stack"|"explicit"|"rule"
);

-- 低置信度/歧义共指项（侧车表，不污染主事实）
CREATE TABLE coref_pending (
    id          SERIAL PRIMARY KEY,
    source_expr TEXT,
    candidates  JSONB NOT NULL,           -- [{"ref_id":"@t","score":0.58}, ...]
    turn_id     INT REFERENCES turns(id),
    status      TEXT DEFAULT 'pending'
);

-- 时间语义
CREATE TABLE temporal_specs (
    stmt_id        TEXT PRIMARY KEY REFERENCES statements(id),
    time_type      TEXT CHECK (time_type IN ('point','interval','fuzzy')),
    resolved_start TEXT,                  -- ISO8601 或 partial
    resolved_end   TEXT,                  -- interval 用
    fuzzy_desc     TEXT,                  -- 原始描述（"recent", "past"）
    window_days    INT,                   -- fuzzy 的近似窗口
    anchor_turn    INT NOT NULL
);
```

### ID 策略

> **设计决议**：每次提取是一次干净独立的上下文，LLM 输出的 `$p1, $p2` 等 ID 只在该批次内有意义。

**`$id`（statements）**：解析器生成全局唯一 ID = `{batch_id}_{local_id}`。例如 LLM 输出 `$p1`，batch_id 为 `b_7f3a`，则实际存储 ID 为 `b_7f3a_p1`。

**`@id`（refs）**：按 `(scope, ref_type, key)` 做 upsert。两次对话都提到 `@local/person("tom")` → 同一行，aliases 合并。`_:blank` 节点加 batch_id 前缀避免冲突。

### 写入规则

每条语法行 → 写入对应表：

| 句式 | 写入 |
|------|------|
| `@id = ...` | `refs` 表 upsert（按 scope+ref_type+key 去重）|
| `$id = pred(args)` | `statements` 表 1 行（id=batch前缀+本地id）+ `stmt_refs` N 行；含内联谓词时先展开；含 `time()` 时同步写 `temporal_specs` |
| `ev($id, ...)` | `evidence` 表 1 行 |
| `note($id, ...)` | `notes` 表 1 行 + 检测 `NEW_PRED` 更新 `vocab_registry`（含碰撞检测） |
| `# ...` | 忽略 |
| 共指消解结果 | `coreference` 或 `coref_pending`（由焦点栈模块写入） |
| `correct_intent` / `retract_intent` | 触发修正流程：查库匹配旧语句 → 标记 `is_current=FALSE` + `superseded_by` |

### 典型查询

```sql
-- "Tom 的职业是什么"
SELECT args->1 FROM statements WHERE predicate='occupation'
  AND id IN (SELECT stmt_id FROM stmt_refs WHERE ref_id='@t' AND position=0);

-- "我所有的朋友"
SELECT sr.ref_id FROM stmt_refs sr JOIN statements s ON sr.stmt_id = s.id
  WHERE s.predicate='friend' AND sr.position=1
  AND s.id IN (SELECT stmt_id FROM stmt_refs WHERE ref_id='@s' AND position=0);

-- "置信度低于 0.5 的记忆"
SELECT s.* FROM statements s JOIN evidence e ON s.id = e.target_id WHERE e.conf < 0.5;

-- "Tom 说过的所有话"
SELECT s2.* FROM statements s1
  JOIN stmt_refs sr ON s1.id = sr.stmt_id AND sr.position = 1
  JOIN statements s2 ON sr.ref_id = s2.id
  WHERE s1.predicate = 'say'
  AND s1.id IN (SELECT stmt_id FROM stmt_refs WHERE ref_id='@t' AND position=0);
```

## 13. 知识图谱映射

| 符号层 | KG 映射 |
|--------|---------|
| `@id` (REF) | **Node** |
| `$p = pred(@a, @b)` (PROP，两个 ref arg) | **Edge** a→b, label=pred |
| `$p = pred(@a, "value")` (PROP, ref+literal) | **Property** on node a |
| `$f = frame(@a, $p)` (FRAME) | **Reified statement node** $f, incoming from @a, links to $p |
| `ev($p, ...)` | **Edge property / provenance** on $p |

所有 `$id` 在 KG 中统一映射为 **reified statement node**，边只表示 arg 位置关系。这等价于 RDF Named Graph 思路。

## 14. 与当前 MIND 系统的关系

### 字段映射（参考对照）

| 语义翻译层 | 当前 FactEnvelope 字段 |
|------------|----------------------|
| `@self` | `subject_scope = "self"` |
| `@local/person("tom")` + `friend(@s,@t)` | `subject_scope = "third_party_named"`, `display_name = "tom"`, `relation_type = "friend"` |
| `_:p1` + `friend(@s,_:p1)` | `subject_scope = "third_party_unknown"`, `relation_type = "friend"` |
| `predicate` | `field_key` |
| arg[1] (literal) | `field_value_json = {"value": ...}` |
| `ev(conf=N)` | `confidence = N` |
| `ev(src=...)` | `source_context` |
| FRAME 系统 | ⚠️ **FactEnvelope 无此能力** |
| QUALIFIER time() | ⚠️ **FactEnvelope metadata 可承载，但不够** |

### 实施策略：直接使用新 Schema

> **设计决议**：废弃 Phase 1 投影回 FactEnvelope 的方案。Phase 1 会丢弃 Frame 信息（believe, hope, say, neg, lie 等），直接违反"不丢信息"的核心目标。

**从第一天起使用本文第 12 节定义的完整表结构**（refs, statements, stmt_refs, evidence, notes, vocab_registry, coreference, coref_pending, temporal_specs）。Frame 全量存储。

渐进策略改为——**表结构一步到位，功能覆盖率分阶段提升**：

| 阶段 | 目标 |
|------|------|
| **阶段 1** | 基础提取：Ref + Prop + Evidence，覆盖事实/属性/关系 |
| **阶段 2** | 完整语义：加入 Frame + Qualifier，覆盖态度/条件/转述/否定 |
| **阶段 3** | 运行时增强：共指消解焦点栈、时间即时解析、词汇碰撞检测 |

## 15. LLM Prompt 模板

```text
You are a semantic extraction engine.
Convert conversations into structured statements using EXACTLY these 5 forms:

  @id = entity_ref                          # declare entity
  $id = predicate(arg, arg, ...)            # assert proposition or frame
  ev($id, conf=N, src="turn_X", span="…")  # attach evidence
  note($id, "explanation")                  # free-text note
  # comment                                 # ignored by parser

Args can be: @id, $id, "literal", number, [list], or pred(args) (2 levels max).

## Entity refs
  @self                              # the owner/user
  @local/TYPE("name")                # owner-local entity (person, animal, object...)
  @local/TYPE("name", alias=[...])   # with known aliases
  @world/TYPE("name")                # world entity (city, brand, org...)
  _:id                               # unnamed entity (blank node)

## Common predicates — use when they fit

PROP (assertions about the world):
  Relations: friend mother father brother sister spouse partner child cousin
    coworker boss mentor student roommate neighbor classmate teammate
    client landlord doctor pet
  Attributes: name age occupation location workplace education nationality
  Actions: like dislike habit hobby skill own use eat drink speak
    live_in work_at study_at
  Events: plan event buy visit meet resign marry move start stop birthday gift

FRAME (attitudes, logic, modality wrapping a proposition):
  Cognitive: believe doubt know uncertain
  Volition: hope want intend
  Speech: say recommend ask promise
  Logic: if cause because
  Deontic: must permit should
  Truth: neg lie joke retract_intent correct_intent
  Emotion: emotion
  Decision: decide defer undecided

QUALIFIER (dimensions that modify another proposition):
  time degree quantity frequency duration location

## Creating new predicates
If no seed predicate fits precisely, CREATE a new one:
- lowercase_snake_case English
- Attach: note($id, "NEW_PRED word | category | arg_schema | definition")
- Prefer specific over vague: obsessed_with > like, bicker > quarrel

## Multi-value attributes
When an entity has multiple values for the same predicate, EXPAND into separate statements:
  $p1 = speak(@s, "中文")
  $p2 = speak(@s, "英语")
Instead of: speak(@s, ["中文", "英语"])

## Corrections and retractions
When the user corrects or retracts a previous statement, express the INTENT:
  $p_new = new_assertion(...)
  $f = correct_intent(@s, $p_new)
  note($f, "CORRECTION: describe what is being corrected")
For retraction:
  $f = retract_intent(@s, "description of what is being retracted")
Do NOT reference $ids from previous extraction batches — you don't have access to them.

## Rules
- One statement per line
- Declare @refs before using them (@self is implicit)
- Max 2 levels of inline nesting; deeper → use intermediate $ids
- Lists use [a, b, c] syntax; no nested lists (prefer expanded form)
- Attach ev() to every $id — minimum conf and src
- Use note() for info that cannot be formalized
- Do NOT output natural language outside these 5 forms
- Do NOT invent facts not stated in the conversation
- Ignore assistant replies unless the user explicitly adopts them

{#   @t = @local/person("tom")    # score=0.82, last mentioned turn 6  #}
{#   @sa = @local/person("sarah") # score=0.65, last mentioned turn 8  #}
{# 如果是首次对话（无历史），此区块为空 #}
```

## 16. 解析器设计

### 行类型识别：5 条正则

```python
import re

PATTERNS = {
    "comment": re.compile(r'^\s*#'),
    "ref":     re.compile(r'^@(\w+)\s*=\s*(.+)$'),
    "prop":    re.compile(r'^\$(\w+)\s*=\s*(\w+)\((.+)\)$'),
    "ev":      re.compile(r'^ev\(\$(\w+)(.+)\)$'),
    "note":    re.compile(r'^note\(\$(\w+),\s*"(.+)"\)$'),
}
```

> **注意**：5 条正则仅用于**判断行类型**。参数拆分（尤其是内联嵌套 `plan(@s, buy(@s, "bookshelf"))`）需要括号匹配栈，纯正则无法正确处理嵌套括号。

### 参数拆分：括号匹配栈

```python
def split_args(args_str: str) -> list[str]:
    """按顶层逗号拆分参数，尊重括号嵌套和字符串引号。"""
    result = []
    current = []
    depth = 0
    in_string = False
    for ch in args_str:
        if ch == '"' and (not current or current[-1] != '\\'):
            in_string = not in_string
        if not in_string:
            if ch in '([':
                depth += 1
            elif ch in ')]':
                depth -= 1
            elif ch == ',' and depth == 0:
                result.append(''.join(current).strip())
                current = []
                continue
        current.append(ch)
    if current:
        result.append(''.join(current).strip())
    return result
```

### 四级级联降级

每行独立经过四级管线，确保**不丢信息**：

```
级别 1 — 严格解析
  行 → 正则判类型 + 括号匹配栈拆参数 → 成功 → 入库(标记 strict)
                                         ↓ 失败
级别 2 — 模糊修复
  自动修正常见错误 → 重试严格解析 → 成功 → 入库(标记 fuzzy)
  修复规则：
  - typo 修正：confirm→conf, souce→src
  - 括号补全：检测不配对括号，自动补齐
  - 引号修正：中文引号 ""→ 英文 ""
  - 字段顺序容错：ev() 内 kv_pair 顺序不固定
                                         ↓ 仍失败
级别 3 — LLM 批量修正
  累积本轮所有失败行 → 一次 LLM 调用修正格式 → 重试 → 成功 → 入库(标记 llm_corrected)
                                         ↓ 仍失败
级别 4 — 降级为 NOTE
  原始文本 → note($nearest_id, "PARSE_FAIL: 原文") → 入库(标记 fallback)
  信息不丢失，降级保全。
```

### REF 行失败特殊处理

如果某 `@id` 的 REF 声明行解析失败，后续引用 `@id` 的语句都会变成悬空引用。处理方式：遇到未声明的 `@id` 时，自动创建 fallback ref（`scope=unknown`），避免连锁失败。

### 关键特性

- **行级独立**：每行独立解析，单行错误不影响其余行（REF 失败有 fallback）
- **括号匹配栈**：正确处理 2 层内联嵌套参数
- **内联谓词自动展开**：内联嵌套是语法糖，解析器自动展开为独立语句（见下）
- **四级容错**：strict → fuzzy → LLM → note，保证信息不丢
- **解析标签**：每条入库记录标记解析级别（strict/fuzzy/llm_corrected/fallback），可追溯
- **ID 全局化**：LLM 输出的本地 `$id` 自动加 `batch_id` 前缀生成全局唯一 ID

### 内联谓词展开

内联嵌套是纯**语法糖**。解析器自动展开为独立 `$id` 语句，存储全部扁平化：

```text
# LLM 输出（语法糖）
$p1 = plan(@s, buy(@s, "bookshelf"))

# 解析器实际存储（展开后）
$_auto1 = buy(@s, "bookshelf")     ← 解析器自动生成
$p1 = plan(@s, $_auto1)            ← args 中的内联替换为 $_auto1 引用
```

展开规则：
- 检测 args 中包含 `pred(...)` 形式的内联谓词
- 为每个内联谓词生成 `$_autoN` 的独立 statement
- 原语句 args 中替换为 `$_autoN` 引用
- 递归处理（最多 2 层），保证所有语句都是扁平的
- `stmt_refs` 反向索引覆盖展开后的所有引用，查询完整

## 17. 共指消解：焦点栈

### 问题

中文对话大量省略主语（零指代），长对话中代词"他/她"的指代也会变得模糊。草稿原方案完全依赖 LLM 自行消解，50+ turn 后 LLM 容易丢失焦点。

### 管线位置：Prompt 注入（方案 A）

> **设计决议**：焦点栈作为 **LLM 提取前的上下文注入**，不是后处理校验。

```
每次提取前：
  1. 从 refs + coreference 表查询当前焦点栈 top-k
  2. 将焦点栈注入 Prompt 的上下文区（见 §15 Prompt 模板）
  3. LLM 在提取时直接使用焦点栈中的 @id，减少消解错误

提取完成后：
  4. 根据本次提取结果更新焦点栈评分
  5. 记录消解事件到 coreference / coref_pending 表
```

注入格式（附加在 Prompt 末尾、对话文本之前）：

```text
## Active entities (focus stack)
The following entities are currently active in conversation context.
Use their @id when they are referenced by pronouns or zero-anaphora:
  @t = @local/person("tom")    # score=0.82, last mentioned turn 6
  @sa = @local/person("sarah") # score=0.65, last mentioned turn 8
  @m = @local/person("mom")    # score=0.51, last mentioned turn 15
```

### 焦点栈数据结构

```python
class FocusEntry:
    ref_id: str           # @id
    ref_type: str         # "entity" | "event"
    score: float          # 综合得分
    last_turn: int        # 最近提及的 turn
    mention_count: int    # 累计提及次数

class FocusStack:
    entries: list[FocusEntry]  # 按 score 降序
    top_k: int = 5             # 活跃窗口
```

### 评分公式（五维加权）

```
S(e) = 0.35*R + 0.20*F + 0.20*G + 0.15*T + 0.10*P

R 近因性 = 0.78^(当前turn - 最近提及turn)
F 频度   = min(1.0, count/10)
G 语法角色 = 主语1.0 / 宾语0.6 / 其他0.3
T 话题相关 = 当前turn与实体的embedding相似度
P 说话人   = 说话人提过+0.1 / 说话人自身+0.2
```

### 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `ambiguity_margin` | 0.12 | top-1 与 top-2 分差 < 此值 → 标记歧义 |
| `auto_accept` | 0.62 | top-1 > 此值 → 直接接受 |
| `explicit_mention_boost` | +0.35 | 显式提及名字的加分 |
| `long_gap_penalty` | -0.15 | turn 间隔 > 8 时惩罚 |

### 中文零指代处理

句子省略主语时的四步级联：

1. **话题链延续** → 前句主语 = 焦点 top-1？直接继承
2. **动词语义约束** → "辞职"需要 human → 过滤焦点栈中非 human 实体
3. **分差判断** → top-1 vs top-2 分差 ≥ margin → 接受；< margin → 标记歧义
4. **全部失败** → 生成 `_:anon` + 写入 `coref_pending` 待消解

### 存储

消解结果写入 `coreference` 表（§12 已定义）。低置信度/歧义项写入 `coref_pending` 侧车表，不污染主事实流。

## 18. 时间语义

### 问题

`time($p1, "next_month")` —— relative to what? 草稿中 `time()` 大量使用相对时间表达，但没有定义参考点、没有区分时间类型。

### 三种时间类型

| 类型 | 语法示例 | 说明 |
|------|---------|------|
| `point` | `time($p, "2026-04")` | 精确到年/月/日 |
| `interval` | `time($p, "2026-04", "2026-06")` | 起止区间 |
| `fuzzy` | `time($p, "recent")` | 模糊描述 |

### 锚点规则

**相对时间的参考点 = 该 turn 的实际时间戳**。

入库时尝试将相对时间转绝对值：

| 原始表达 | turn 时间 | 解析结果 |
|---------|----------|----------|
| `"next_month"` | 2026-03-30 | `resolved_start = "2026-04"` |
| `"last_year"` | 2026-03-30 | `resolved_start = "2025"` |
| `"recent"` | 2026-03-30 | `fuzzy_desc = "recent"`, `window_days = 30` |
| `"past"` | 2026-03-30 | `fuzzy_desc = "past"`, 不设 window |

### 存储

解析后写入 `temporal_specs` 表（§12 已定义）。`time()` qualifier 仍然作为 `statements` 表中的语句存在，`temporal_specs` 是其结构化补充。

## 19. 版本控制与修正

### 问题

用户会修正之前的说法（"不对，Tom 其实没辞职，他只是在犹豫"）。需要标记旧信息被取代，而不是物理删除。

### 设计约束

每次提取是干净独立的上下文，LLM **不知道**数据库中已有语句的 `$id`。因此版本控制不能靠 LLM 在输出中直接引用旧 `$id`。

### 修正流程（AI 主导查询）

```
1. LLM 提取当前对话 → 产生新语句（例如 $p_new = uncertain(@t, resign(@t))）
   同时 LLM 标记这是一个修正/撤回意图：
   $f = correct_intent(@s, $p_new)
   note($f, "CORRECTION: 用户修正了之前关于 Tom 辞职的说法")

2. 解析器检测到 correct_intent / retract_intent → 触发修正流程

3. 修正流程：
   a. 用 note 中的语义描述 + $p_new 的 predicate/args 查询数据库
   b. 找到 top-k 最相关的已有语句（按 predicate + ref 重叠度 + 时间排序）
   c. AI 判断是否匹配某条旧语句 → 确认后标记:
      - 旧语句 is_current = FALSE
      - 旧语句 superseded_by = $p_new.id
      - $p_new is_current = TRUE

4. 如果找不到匹配的旧语句 → $p_new 作为独立新语句入库，不做修正关联
```

### LLM 输出格式

LLM 不需要知道旧 `$id`，只需表达修正意图：

```text
# 用户说："不对，Tom 其实没辞职，他只是在犹豫"
@t = @local/person("tom")
$p1 = uncertain(@t, resign(@t))
$f1 = correct_intent(@s, $p1)
note($f1, "CORRECTION: 修正之前关于 Tom 计划辞职的说法，实际上 Tom 只是在犹豫")
ev($p1, conf=0.9, src="turn_15")
ev($f1, conf=0.9, src="turn_15", span="不对...其实没辞职")
```

### 解析器行为

1. 检测到 `correct_intent` 或 `retract_intent` 谓词
2. 提取 note 中的 `CORRECTION:` 描述 + 关联语句的语义信息
3. 查询 `statements` 表：`WHERE predicate IN (相关谓词) AND is_current = TRUE`，并通过 `stmt_refs` 过滤相关实体
4. 对候选结果排序，取 top-1 → 如果置信度够高，自动关联；否则记录到待确认队列

### 查询约定

- 默认查询加 `WHERE is_current = TRUE`
- 历史回溯查询可移除此条件，查看完整修改链
- 通过 `superseded_by` 可追溯版本演进

## 20. 已知边界与不追求的事项

### 覆盖范围估计

> **注**：以下覆盖率为设计分析估计，未经真实 LLM 输出验证。需用真实对话 + 真实 LLM 生成来测。

| 信息层次 | 覆盖率 | 说明 |
|---------|--------|------|
| 事实/属性/关系 | ~95% | 列表值展开后基本完备 |
| 态度/意图/义务 | ~90% | Frame 系统覆盖 |
| 逻辑/推理 | ~70% | if/cause/neg 覆盖主要场景 |
| 元交互 | ~60% | retract/correct/joke 覆盖常见修正 |
| 共指消解 | 新增 | 焦点栈 + 零指代规则（§17） |
| 时间语义 | 新增 | 三类型 + 锚点即时解析（§18） |
| 版本控制 | 新增 | retract/correct + is_current（§19） |
| 无法形式化的残余 | 100% | note() 兜底 |

### 第一版不追求

| 不追求 | 原因 |
|--------|------|
| 完整一阶逻辑（∀/∃ 量化） | 日常对话极少出现严格量化 |
| 多层嵌套量词 | 复杂度激增，LLM 表达不稳定 |
| 完整时态逻辑 | time() qualifier + "past"/"now"/"fut" 已够用 |
| 全局实体消歧 | 由应用层通过 alias 匹配和 embedding 相似度解决 |
| 严格的逻辑完备性 | 目标是信息保全，不是定理证明 |

### 瓶颈不在符号层

长对话提取的真正瓶颈在 LLM 的 3 个能力上：

| LLM 能力瓶颈 | 说明 |
|-------------|------|
| 跨 turn 共指消解 | 焦点栈（§17）缓解，但极端长对话仍依赖 LLM |
| 置信度判断 | "好像" vs "确实" 的区分 |
| 噪声过滤 | 助手的回复不是用户事实 |

建议对超长对话（>50 turns）**分段提取 + 合并去重**。符号层天然支持这种模式——每段独立输出，后续按 `@id` / alias 合并同一实体。
