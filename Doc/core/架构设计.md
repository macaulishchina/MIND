# MIND 架构设计

## 1. 设计原则

1. 用 mem0 的流程骨架，填我们自己的质量内核
2. LLM 做决策引擎，不写规则引擎
3. 数据模型分两层：核心层保证简洁，增强层保留天花板
4. 先做最小闭环，不做平台化过度设计
5. 组件可替换，但不追求支持所有后端

## 2. 系统架构

### 总体流程

```
用户输入
  → Memory.add()
    → LLM 提取事实
    → 搜索相似旧记忆（向量检索）
    → LLM 判断 ADD / UPDATE / DELETE / NONE
    → 执行操作（写入/更新/删除）
    → 记录操作历史

用户提问
  → Memory.search()
    → Embedding 编码查询
    → 向量检索 + 元数据过滤
    → 可选：重排
    → 返回相关记忆
    → 注入回答上下文
```

### 与 mem0 的差异点

在 mem0 基线流程之上，叠加三个增强：

1. **写入增强**：LLM 提取事实时同时输出 confidence，低置信记忆标记为候选状态
2. **更新增强**：UPDATE 操作保留旧版本引用（version_of 字段），不直接覆盖
3. **检索增强**：v1 先做基础向量检索，v1.5 引入时间衰减权重

## 3. 模块设计

### 3.1 Memory 主类

系统唯一入口。不引入独立的"Memory Controller"概念——Memory 类本身就是控制器。

核心接口：

```
add(messages, user_id, metadata) → 提取并存储记忆
search(query, user_id, limit, filters) → 检索相关记忆
get(memory_id) → 获取单条记忆
get_all(user_id, limit) → 获取全部记忆
update(memory_id, data) → 手动更新记忆
delete(memory_id) → 删除记忆
history(memory_id) → 查看变更历史
```

### 3.2 LLM 层

职责：
- 从对话中提取事实
- 判断新旧记忆的关系（ADD/UPDATE/DELETE/NONE）
- 为事实标注置信度

v1 至少支持两种 LLM 后端（如 OpenAI + Ollama），通过 Factory 切换。

### 3.3 Embedding 层

职责：
- 对记忆内容和查询进行向量编码

v1 至少支持两种 Embedding 后端，通过 Factory 切换。

### 3.4 存储层

职责：
- 向量存储（记忆内容 + embedding + 元数据）
- 历史记录（操作日志）

v1 推荐：
- 开发/测试：FAISS 或 Qdrant in-memory（零依赖）
- 生产：PostgreSQL + pgvector 或 Qdrant

历史记录用 SQLite 即可（参考 mem0 做法）。

## 4. 数据模型

### 两层设计

#### 核心层（v1 必须，运行时依赖）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 记忆唯一标识 |
| user_id | string | 所属用户 |
| content | string | 记忆内容 |
| hash | string | MD5 去重 |
| metadata | dict | 自由元数据 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

#### 增强层（v1 有字段但可不填，v1.5+ 逐步启用）

| 字段 | 类型 | 说明 | 启用时机 |
|------|------|------|---------|
| confidence | float | 置信度（0-1） | v1：LLM 提取时标注 |
| status | enum | active / deleted | v1 只用两个状态 |
| source_context | string | 原始对话片段 | v1：写入时记录 |
| source_session_id | string | 来源会话 | v1：写入时记录 |
| version_of | string | 指向被更新的旧记忆 ID | v1：UPDATE 时记录 |
| importance | float | 重要性 | v1.5：引入衰减时启用 |
| type | enum | profile / preference / event | v1.5：引入类型感知检索时启用 |

#### 设计理由

- 核心层对齐 mem0 简洁度，7 个字段就能跑通闭环
- 增强层的字段从第一天就存在于数据模型中，避免后续数据迁移
- 但 v1 的检索和更新流程**不依赖**增强层字段
- confidence 和 source_context 在 v1 就开始写入，为后续质量分析积累数据

### 状态模型

v1 只用两个状态：

```
active ←→ deleted
```

v2 扩展为：

```
active → superseded（被新版本替代）
active → inactive（暂时不参与召回）
active → deleted（逻辑删除）
```

### 版本追踪

v1 中 version_of 只是一个记录字段，不构建链式查询：

```json
{
  "id": "mem_002",
  "content": "用户现在只喝美式",
  "version_of": "mem_001",
  "status": "active"
}
```

旧记忆 mem_001 在 v1 中仍为 active（两个版本并存），在 v2 中才引入 superseded 状态。

## 5. 技术选型

### v1 推荐技术栈

| 组件 | 推荐 | 备选 |
|------|------|------|
| 语言 | Python | - |
| LLM | OpenAI GPT-4o-mini | Ollama 本地模型 |
| Embedding | OpenAI text-embedding-3-small | HuggingFace 本地 |
| 向量存储 | Qdrant（in-memory 开发，server 生产） | PostgreSQL + pgvector |
| 历史记录 | SQLite | - |

### 为什么不用 PostgreSQL 作为 v1 底座

原定方案是 PostgreSQL + pgvector 作为唯一底座。调整理由：

1. 开发环境搭建成本高，拖慢第一个闭环
2. mem0 证明了向量库 + SQLite 的组合完全够用
3. PostgreSQL 的事务优势在 v1 阶段用不到（单用户、无复杂事务）
4. 通过 Factory 模式，后续随时可以切到 PostgreSQL

PostgreSQL + pgvector 保留为生产部署推荐方案，但不是 v1 开发的前置依赖。

## 6. 技术演进路线

### v1：最小闭环

- add / search / update / delete 四个核心操作
- LLM 两步决策（提取 + 操作判断）
- 基础向量检索
- 写入时记录 confidence 和 source_context（积累数据）
- version_of 字段记录更新关系

### v1.5：质量增强（第一个差异化点）

- 基于 confidence 的召回降权
- 基于时间的衰减权重
- 低置信记忆的渐进确认机制
- 简单的记忆合并（相同主题去重）

### v2：检索增强 + 生命周期

- 状态机完整化（superseded / inactive）
- 混合检索（向量 + 元数据过滤）
- 查询分解（多子查询召回）
- Reranker
- 后台巩固任务

### v3：关系增强

- 图数据库（Neo4j 或 Kuzu）
- 实体关系和事件链
- 记忆压缩与抽象保留
- 更强的选择性遗忘

### 触发升级的信号

| 从 | 到 | 信号 |
|---|---|------|
| v1 | v1.5 | v1 跑通闭环，开始积累真实数据 |
| v1.5 | v2 | 简单衰减不够，需要更精细的检索和生命周期 |
| v2 | v3 | 需要表达事件关系和因果链 |

## 7. 与 mem0 的架构差异总结

| 维度 | mem0 | MIND |
|------|------|------|
| 数据模型 | 7 字段极简 | 7 核心 + 7 增强，分层启用 |
| 写入质量 | LLM 一次性判断 | LLM 判断 + confidence 标注 |
| 更新策略 | 直接覆盖 | 覆盖但保留 version_of |
| 遗忘机制 | 无 | v1.5 引入时间衰减 |
| 检索策略 | 纯向量 + 可选重排 | v2 引入查询分解和多路召回 |
| 状态管理 | 无状态机 | v1 两状态，v2 完整状态机 |
| 后端支持 | 27 种向量库 | 2-3 种，通过 Factory 切换 |
