# MIND 评测方案

## 1. 目标

验证两件事：

1. 当前 MVP 公共 API 是否真正形成闭环
2. 当前 STL-native add 主链在评测入口上是否可被复现和观察

## 2. 评测方式

当前采用三层验证：

1. **基础自动化回归**：`pytest tests/`
2. **公共 API 烟测**：`tests/test_memory.py` 中的 MVP smoke path
3. **阶段级评测**：统一入口 `tests/eval/runners/eval_cases.py`

## 3. MVP 必测场景

### 场景 1：稳定偏好写入与召回

步骤：
1. 用户输入"我喜欢黑咖啡"
2. 新会话，询问"你推荐我喝什么"

验证项：
- [ ] 偏好记忆被正确写入
- [ ] `search()` 能召回该偏好
- [ ] 记忆包含 confidence 和 source_context

### 场景 2：偏好更新与版本追踪

步骤：
1. 已存在"喜欢黑咖啡"
2. 用户输入"我现在只喝美式"
3. 再次检索该偏好相关问题

验证项：
- [ ] 新记忆写入成功
- [ ] 新记忆的 version_of 指向旧记忆
- [ ] active 检索面优先看到新值
- [ ] 通过 history() 可查看变更记录

### 场景 3：公共 API 手动更新路径

步骤：
1. 已存在一条 memory
2. 调用 `update(memory_id, content)`
3. 调用 `get()` 和 `history()`

验证项：
- [ ] 内容更新成功
- [ ] `history()` 包含 UPDATE 记录

### 场景 4：失效记忆不再污染

步骤：
1. 删除一条记忆
2. 调用 `search()` 和 `get_all()`

验证项：
- [ ] 该记忆不出现在检索结果中
- [ ] 该记忆不出现在 active `get_all()` 结果中
- [ ] `history()` 包含 DELETE 记录

## 4. 维护中的验证命令

### 4.1 全量回归


### 4.2 仅看 Memory 公共链路

```bash

  --stage owner_add \
  --toml mindt.toml \

```

### 4.4 MVP Live Owner-Add 基线

```bash
python tests/eval/runners/eval_cases.py \
  --stage owner_add \
  --toml mind.toml \
  --pretty \
  --output tests/eval/reports/mvp_live_owner_add_baseline_YYYY-MM-DD.json
```

说明：
- 这一步用于记录真实模型下的 point-in-time 基线，不替代 `pytest tests/`
- 当前已归档的一次基线记录位于
  `.ai/archive/mvp-live-eval-baseline/artifacts/owner_add_live_baseline_2026-04-02.md`
- 2026-04-02 基线结果：
  - `canonical_text_accuracy = 0.667`
  - `subject_ref_accuracy = 0.667`
  - `count_accuracy = 0.714`
  - `owner_accuracy = 1.000`
  - `case_pass_rate = 0.643`
- 这一轮基线表明：当前默认在线 owner-add 路径的主要短板在 self anchoring、纠正投影、多轮中文综合 case 和时间限定词附着

## 5. 质量观测指标

以下指标在 MVP 阶段**不作为日常通过标准**，但需要记录数据以指导 v1.5 优化方向。

### 4.1 写入质量（对应核心问题一）

| 指标 | 定义 | 观测方法 |
|------|------|---------|
| 提取准确率 | 提取的事实中真正正确的比例 | owner-add / stl_extract 人工抽检 |
| 噪声写入率 | 不该记录的信息被错误写入的比例 | 人工抽检 |
| confidence 校准度 | confidence 高的记忆是否确实更准确 | 按 confidence 分桶观察 |

### 4.2 去重质量

| 指标 | 定义 | 观测方法 |
|------|------|---------|
| 重复率 | 同一事实被存为多条记忆的比例 | 人工检查样本用户的 active memories |
| UPDATE 准确率 | 应该判 UPDATE 的是否正确判了 | owner-add case + 人工抽检 |

### 4.3 检索质量（对应核心问题三）

| 指标 | 定义 | 观测方法 |
|------|------|---------|
| 召回命中率 | 需要的记忆是否在 Top-K 中 | 公共 smoke path + 人工构造查询 |
| 任务相关性 | Top-K 中与当前任务真正相关的比例 | 人工评判 |

## 6. 与 mem0 的对比评测

在相同测试场景下，对比 MIND v1 和 mem0 的表现。

### 对比维度

| 维度 | 评测方法 |
|------|---------|
| 提取准确率 | 同一组对话，分别用两个系统提取，人工比对 |
| 重复率 | 运行 50 轮对话后，人工统计重复记忆数量 |
| 可追溯性 | 随机选 5 条记忆，能否追踪到来源对话 |
| 更新正确性 | 10 个偏好变更场景，对比更新结果 |

### 预期差异

| 维度 | mem0 预期 | MIND 预期 |
|------|----------|----------|
| 核心流程 | 可用 | 与 mem0 持平或更稳 |
| 可追溯性 | 仅有 SQLite 日志 | confidence + source_context + version_of + history |
| 遗忘能力 | 无 | v1 无，v1.5 引入 |

MVP 阶段 MIND 与 mem0 在核心流程上预期持平。差异化从 v1.5 开始显现。

## 7. 长期评测方向（v1.5+）

### 对应核心问题一：写入质量

- confidence 降权后的召回准确率变化
- 渐进确认机制的噪声过滤效果

### 对应核心问题二：长期健康

- 运行 N 轮后的记忆污染率
- 时间衰减引入前后的检索质量对比
- 合并压缩后的信息保留率

### 对应核心问题三：任务感知检索

- 查询分解引入前后的召回命中率
- 多路召回 vs 单向量召回的任务完成率
