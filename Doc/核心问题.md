# MIND 核心研究问题

## 1. 文档目标

MIND 的差异化价值来自解决三个现有系统不解决的问题。本文档定义这三个问题的本质、当前困境和研究方向。

解决任何一个，就能形成真正的技术差异化。

## 2. 问题一：记忆提取错误的检测与修正

### 问题本质

现有系统（如 mem0）用 LLM 从对话中提取记忆，一次性判断，无验证机制。实际场景中 LLM 会：

- 把他人信息记成用户信息（"帮同事订黑咖啡" → 误记为"用户喜欢黑咖啡"）
- 忽略时态和转折（"以前喜欢跑步，现在膝盖不行了" → 误记为"喜欢跑步"）
- 把假设当事实（"如果去日本可能学日语" → 误记为"想学日语"）

一旦错误记忆写入，在 mem0 体系中永远不会被自动修正——除非未来某次对话恰好产生矛盾触发 DELETE。

### 为什么现有系统不解决

增加写入复杂度，降低"开箱即用"体验。商业产品的客户不会为"少记错几条"付额外的钱。

### 研究方向

1. **置信度标注**：写入时 LLM 同时输出 confidence，低置信记忆在召回时降权
2. **来源上下文保留**：记录原始对话片段，允许事后重新评估
3. **渐进确认**：单次提及为"候选"，多次出现才升级为"稳定记忆"
4. **主动验证**：在合适的对话时机向用户确认不确定的记忆

### 问题分类

| 方向 | 类型 | 说明 |
|------|------|------|
| 置信度标注 | 工程问题 | prompt 设计 + 数据模型 |
| 来源保留 | 工程问题 | 数据模型扩展 |
| 渐进确认 | 工程 + 研究 | 阈值如何定？多少次算"多次"？ |
| 主动验证 | 研究问题 | 何时问？怎么问才不打扰用户？ |

## 3. 问题二：记忆的长期健康（遗忘机制）

### 问题本质

mem0 的记忆只增不减。运行半年后：

```
系统中 2000 条记忆
├── 30% 仍然有效
├── 40% 已过时但没被显式矛盾过（"用户在准备X考试" — 半年前的事了）
├── 20% 同一事实的不同表述（"喜欢美式" / "偏好美式咖啡" / "常喝美式"）
└── 10% 当初就记错了
```

搜索时 top-5 召回，可能一半是垃圾。回答质量随时间退化。

### 为什么现有系统不解决

遗忘意味着可能"忘掉用户认为重要的东西"，商业产品承受不起投诉风险。不做遗忘是最安全的产品决策。

### 研究方向

1. **时间衰减**：长期未被召回的记忆自动降权
   - 难点：衰减曲线怎么定？重要的长期偏好（如过敏信息）不该衰减
2. **合并压缩**：同一主题的多条记忆合并为一条
   - 难点：合并可能丢细节或引入错误
3. **分层保留**：细节可以遗忘，抽象保留
   - 例："喜欢黑咖啡" + "改喝美式" + "尝试冷萃" → 压缩为"咖啡偏好演化：黑咖啡→美式→冷萃"
4. **主动确认**：定期挑选低活跃记忆，在合适时机向用户确认是否仍有效

### 问题分类

这是一个真正的研究问题。没有现成公式能给出正确的衰减曲线。衰减太快会丢重要偏好，太慢等于没衰减。合并压缩需要 LLM 判断，引入新的出错可能。

### 参考方向

- FadeMem（https://arxiv.org/abs/2601.18642）：基于遗忘曲线的记忆衰减
- AgeMem（https://arxiv.org/abs/2601.01885）：基于记忆年龄的管理策略

## 4. 问题三：任务感知检索

### 问题本质

现有系统的检索是 `embed(query) → cosine similarity → top-k`。"语义相似"不等于"当前任务需要"。

```
用户问: "帮我规划下周的行程"

语义相似度召回：
1. "用户上个月去了杭州"          ← 相似但无用
2. "用户喜欢高铁出行"            ← 有用
3. "用户讨论过行程规划工具"       ← 相似但无用
4. "用户下周三有牙医预约"         ← 不相似但关键
5. "用户上周规划了读书计划"       ← 相似但无用
```

第 4 条最关键，但"牙医预约"和"行程规划"语义不相似，可能排不进 top-5。

### 为什么现有系统不解决

需要额外 LLM 调用来理解任务意图，增加延迟和成本。通用平台无法为每种任务定制检索策略。

### 研究方向

1. **查询分解**：把用户问题拆成多个子查询
   - "规划下周行程" → "下周时间安排" + "出行偏好" + "下周已有约定"
2. **类型感知检索**：不同类型的问题去不同的记忆分区找
   - 行程问题 → 优先找时间相关记忆
3. **任务模板**：常见任务类型预定义需要召回哪几类记忆
4. **多路召回合并**：多个检索路径的结果合并后重排

### 问题分类

| 方向 | 类型 | 说明 |
|------|------|------|
| 查询分解 | 工程问题 | 一次额外 LLM 调用 |
| 类型感知 | 工程问题 | 元数据过滤 |
| 任务模板 | 工程 + 研究 | 模板怎么定？怎么泛化？ |
| 多路合并重排 | 研究问题 | 怎么合并不同路径的分数？ |

## 5. 三个问题的关系与优先级

```
问题一（写入质量）→ 决定系统里的记忆"对不对"
问题二（长期健康）→ 决定系统"能不能用半年以上"
问题三（检索质量）→ 决定系统"回忆的东西有没有用"
```

建议优先级：**问题一 > 问题二 > 问题三**

理由：如果记忆本身是错的，检索再精准也没用。如果系统半年就污染了，也没机会做复杂检索优化。

## 6. 记忆质量的完整框架

三个核心问题之外，记忆质量的提升还依赖以下支撑路径（按优先级排列）：

1. **记忆表示升级**（工程基础）
   - 结构化 MemoryItem，让记忆可更新、可追溯、可治理
   - 这是三个核心问题的共同基础

2. **反馈评测闭环**（长期优化）
   - 写入质量评测、召回质量评测、遗忘效果评测
   - 没有反馈，系统无法从"能运行"进化到"越用越稳"

3. **经验抽象**（远期方向）
   - 从多次经历中总结模式，形成更高层经验
   - 属于"让 AI 会成长"的范畴，当前不作为主线

## 7. 远期硬边界

当前路线即使三个问题都解决了，仍然存在硬上限：

- 真正的模型内生长期记忆（需要持续学习）
- 稳定的高阶经验抽象（需要 reflection + distillation）
- 类人级经验迁移（需要跨域泛化能力）

这些属于远期研究方向，不影响当前三个问题的价值。当前路线的现实天花板是：一个**强实用型、可治理、可更新、可解释的长期外部记忆系统**。这已经足够支撑个人长期助手、多会话任务 agent 等场景。
