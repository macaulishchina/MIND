## 各步骤详细说明

### 步骤 1：需求定义（人）

- 在 GitHub Issue 中清晰描述需求
- 包含：要实现什么、为什么需要、验收标准
- **不要**描述具体实现方式——那是 AI 的工作

### 步骤 2：任务拆解（人 + AI 协作）

- 让 AI 帮你将需求拆分为可执行的小任务
- 每个任务应该是：
  - 范围明确（涉及 1-3 个文件）
  - 可独立验证（有明确的通过标准）
  - 足够小（AI 在小范围内更可靠）
- 标记每个任务涉及的模块和规则文件

### 步骤 3：AI 编码

- AI 自动读取 `.ai/CONSTITUTION.md`
- AI 按路由表加载相关 `rules/*.md`
- AI 按变更类型加载 `checklists/*.md`
- AI 生成代码 + 测试 + 文档更新

**给 AI 的指令示例**：
```
请为 MIND 项目新增一个 /v1/tags 接口，支持对 memory object 添加标签。
请先阅读 .ai/CONSTITUTION.md 了解项目规则。
```

### 步骤 4：本地验证（自动化）

```bash
uv run ruff check mind/ tests/ scripts/     # 代码风格
uv run mypy mind/ tests/ scripts/            # 类型检查
uv run pytest tests/ -x                      # 单元测试
```

如果这三个命令全部通过，代码基本质量达标。

### 步骤 5：代码审查（人为主）

审查清单：
- [ ] AI 是否遵守了 CHANGE_PROTOCOL？（改了 service 是否也更新了 registry、test、docs？）
- [ ] 架构分层是否正确？（没有上层导入下层吗？）
- [ ] 逻辑是否正确？（尤其关注边界条件）
- [ ] 有没有过度工程化？（不需要的抽象、不需要的配置项）
- [ ] 有没有安全问题？（SQL注入、硬编码密钥、不安全的默认值）

### 步骤 6：CI 验证（自动化）

GitHub Actions 自动运行：
- `product-readiness.yml`：产品就绪门禁
- `ai-governance.yml`：AI 治理检查（变更协议遵守检测）

### 步骤 7：合并部署（人审批）

- 确认 CI 全绿
- 确认 Code Review 通过
- Merge PR → 自动部署（如配置了 CD）

### 步骤 8：反馈闭环

- 检查 AI 这次编码中有没有犯错
- 如果有规则盲区，更新 `.ai/` 规则
- 记录到 `.ai/health/drift-log.md`
- 定期运行 `scripts/ai_health_check.py` 检查项目健康度

## 紧急修复流程（快速通道）

对于紧急 bug：
1. 描述症状给 AI → AI 定位问题
2. AI 生成修复 + 回归测试
3. 本地验证通过
4. 直接 merge（但事后补审查和 drift-log 记录）

---

## 关于任务粒度的建议

| 任务规模 | 建议 |
|---------|------|
| 改一个函数 | 直接让 AI 做，一次完成 |
| 新增一个 endpoint | 分 2-3 步：service → endpoint → test+docs |
| 新增一个完整功能 | 拆成 5-10 个独立任务，逐个完成 |
| 大规模重构 | 仔细规划，拆成 10+ 个小步骤，每步验证 |

**原则**：AI 在任何单次任务中触及的文件不应超过 5 个。超过了就需要拆分。
