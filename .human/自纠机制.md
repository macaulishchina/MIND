# 自纠机制

> 让项目具备自我发现和修复偏差的能力

---

## 自纠的三个层次

### 层次 1：自动化检查（机器执行）

**每次 PR / 每次 commit**：
- `ruff check` — 代码风格零违反
- `mypy` — 类型检查零错误
- `pytest` — 测试全部通过
- `ai-governance` CI — 变更协议遵守检测

**每天 / 定期**：
- `scripts/ai_health_check.py` — 项目健康度全扫描
- 输出 `.ai/health/latest-report.json`

### 层次 2：人工审查（定期执行）

**每周**：
- 查看 `drift-log.md` 新增条目
- 识别重复出现的 AI 犯错模式
- 更新规则覆盖新发现的盲区

**每月**：
- 规则全量回顾——删除不再适用的规则
- 检查 `CONSTITUTION.md` 是否仍在 500 行以内
- 检查架构决策记录是否仍反映当前代码

### 层次 3：系统性迭代（按需执行）

当发现系统性问题时：
- 运行 `scripts/ai_health_check.py` 获取基线
- 分析健康报告，识别退化趋势
- 制定改进计划
- 执行改进
- 再次运行健康检查，对比改进效果

## 健康度指标

| 指标 | 目标 | 含义 |
|------|------|------|
| ruff 违反数 | 0 | 代码风格无偏差 |
| mypy 错误数 | 0 | 类型系统完整 |
| 测试通过率 | 100% | 功能无回归 |
| drift-log 月增条目 | 趋势递减 | 规则体系越来越完善 |
| .ai/ 文件总行数 | 不无限增长 | 规则保持精简 |

## 反馈闭环图

```
AI 编码
  ↓
自动化检查（ruff + mypy + pytest + CI）
  ↓ 发现问题？
  ├─ Yes → 修复 → 记录 drift-log → 更新规则
  └─ No  → 人工审查
              ↓ 发现问题？
              ├─ Yes → 修复 → 记录 drift-log → 更新规则
              └─ No  → 合并
                        ↓
                     定期健康检查
                        ↓ 退化？
                        ├─ Yes → 系统性迭代
                        └─ No  → 继续
```

## drift-log 的使用

每次发现 AI 的规则违反或盲区，按以下格式记录到 `.ai/health/drift-log.md`：

```markdown
### 2026-03-15 — AI 忘记更新 registry

- **类型**: violation
- **规则**: CHANGE_PROTOCOL.md §2 (Adding a New App Service)
- **详情**: AI 创建了新的 service 文件但没有在 AppServiceRegistry 中注册
- **解决**: 手动修复。考虑在 CONSTITUTION.md 中加粗 registry 注册要求。
```

这些记录是规则迭代的最重要输入。没有 drift-log，规则改进就是盲目的。

## 健康检查脚本

```bash
# 运行全量健康检查
uv run python scripts/ai_health_check.py

# 查看结果
cat .ai/health/latest-report.json
```

脚本会检查：
- ruff 违反数
- mypy 错误数
- 测试结果摘要
- `.ai/` 文件完整性
- 与基线的对比
