# 规则维护指南

> 如何维护 `.ai/` 规则体系，让它随项目进化

---

## 核心原则

1. **规则即代码**：`.ai/` 文件就像代码一样，通过 Git 管理，通过 PR 变更
2. **持续迭代**：规则不是写一次就完了，需要根据实际使用不断调整
3. **少即是多**：每条规则都有认知成本。删除无效规则比新增规则更重要
4. **具体可执行**：AI 需要能直接"执行"的规则，不是原则性的建议

---

## 规则层级

```
.ai/CONSTITUTION.md     ← L0：全局宪法（始终加载，<500行）
.ai/ARCHITECTURE.md     ← 架构参考（按需查阅）
.ai/CONVENTIONS.md      ← 编码约定参考（按需查阅）
.ai/CURRENT_STATE.md    ← 项目状态（每次大变更后更新）
.ai/CHANGE_PROTOCOL.md  ← 变更协议（改代码时必读）
.ai/rules/*.md          ← L1：模块级规则（按路由表加载）
.ai/checklists/*.md     ← L2：变更清单（按任务类型加载）
.ai/health/*            ← 健康度数据（自动生成 + 人工记录）
```

---

## 什么时候更新规则

| 信号 | 动作 |
|------|------|
| AI 犯了同一个错 2 次以上 | 新增或强化对应规则 |
| AI 遵守了一条规则但结果不好 | 修改规则 |
| 规则与实际代码模式不一致 | 更新规则以反映当前代码 |
| 新增了重要的架构模式 | 更新 `ARCHITECTURE.md` |
| 项目进入新 Phase | 更新 `CURRENT_STATE.md` |
| 代码模式发生变化 | 更新 `CONVENTIONS.md` |
| 新增了依赖同步关系 | 更新 `CHANGE_PROTOCOL.md` |
| 旧规则不再适用 | **删除**旧规则 |

---

## 更新流程

1. 在 `drift-log.md` 中记录发现的问题
2. 确定需要更新哪个规则文件
3. 修改规则文件（像改代码一样提 PR）
4. Review 规则变更（确保不与其他规则冲突）
5. Merge 后验证 AI 在新规则下的行为

---

## 规则质量检查清单

- [ ] 规则是具体的、可执行的（不是"注意质量"而是"文件 < 500 行"）
- [ ] 规则有唯一的位置（不在多个文件中重复）
- [ ] `CONSTITUTION.md` 未超过 500 行
- [ ] 每个 `rules/*.md` 未超过 200 行
- [ ] 路由表（`CONSTITUTION.md` §6）覆盖了所有主要目录
- [ ] `CHANGE_PROTOCOL.md` 覆盖了所有常见变更类型
- [ ] 不存在相互矛盾的规则

---

## 关键度量

通过 `scripts/ai_health_check.py` 跟踪：
- **ruff 违反数**：MUST 为 0
- **mypy 错误数**：MUST 为 0
- **测试通过率**：MUST 为 100%
- **drift-log 新增条目数**：趋势应该递减
- **规则文件总行数**：不应无限增长

如果 drift-log 条目持续增加，说明规则体系有系统性盲区需要解决。
如果规则总行数持续增长，说明需要做一次规则精简。
